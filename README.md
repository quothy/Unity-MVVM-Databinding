# Unity-MVVM-Databinding
An [MVVM](https://en.wikipedia.org/wiki/Model%E2%80%93view%E2%80%93viewmodel) data-binding &amp; theming package for Unity 2021 and higher.

## At A Glance

![MVVMDatabinding_SampleProject](https://github.com/user-attachments/assets/65a65b9b-2e7b-4677-a09d-f3fb800ecfd2)

This package provides a workflow-focused data binding and theming solution for UI built in Unity's Canvas system. Data is bound to the UI via user friendly Inspector-based workflows, while the developer workflow makes it simple and fast to add new data for binding. Particular emphasis is placed on runtime performance and the ability to work in parallel within mixed discipline teams.

Theming is built on top of the data binding system and offers a no code workflow for styling your UI. It has built-in support for authoring Light, Dark, and High Contrast variants for your UI theme and offers near-instantaneous swapping at runtime.

## Data Binding

The core functionality of this framework is rather straightforward: data consumers can subscribe to updates and data sources can publish those updates. The tricky part is how to ensure sources and consumers have a shared understanding of what data is available and at what address it can be subscribed or published. The vast majority of the power and benefits of this particular Unity MVVM implementation are found within the workflow rather than anything particular clever in the core functionality.

This section of the documentation is split into two focuses: **publishing** and **consuming**. The sub-sections will be wholly concerned with their area of the problem, as they are abstracted away from each other and are agnostic to each other.

### Publishing Data

The first step to publishing data for the UI to consume is to create a `ViewModel`. This framework provides two base classes to build upon:
* `BaseGlobalViewModel`: Choose this class when you want to create a View Model that will only ever have one instance publishing data at a time. 
* `BaseLocalViewModel`: Choose this class in any other circumstance, though there will be an assumption that the View Model component is available in the hierarchy relative to the consumers. Local View Models have extra logic built around them to ensure that we are publishing and subscribing to the correct instance of the View Model at runtime.

For example purposes, let's say we want to build a local View Model to populate data about `FooBar`. The shell of the class will look something like this:

```
using MVVMDatabinding;

public partial class FooBarViewModel : BaseLocalViewModel
{
    
}
```
The class must be declared as `partial` because there will be code automatically generated to take care of some repetitive boilerplate code that would otherwise need to be handwritten. 

Next, let's add data to `FooBarViewModel`. We need to be able to indicate the `FooBar` state, a counter representing `Foo` and a counter representing `Bar`. We also would like the user to be able to click buttons that increment the counters for `Foo` and `Bar`. When the sum of the counters is divisible by 5, then the `FooBar` state is true.

Bindable data items are expressed as properties with public getters and setters. All of them should call `OnPropertyChanged()` within the setter if the value has changed. There is also a special type of bindable data called a Bindable Action, which can only be used on methods with a `void()` signature.

Note: This may seem like code that feels like boilerplate and should be autogenerated, so why isn't it? Well, maybe it should be! But for now, there are enough instances where we want to perform some custom action when the value changes that it hasn't yet been made boilerplate.

```
using MVVMDatabinding;

public partial class FooBarViewModel : BaseLocalViewModel
{
    private bool isFooBar = false;
    [BindableData(0)]
    public bool IsFooBar
    {
      get => isFooBar;
      set { if (isFooBar != value) { isFooBar = value; OnPropertyChanged(); } }
    }

    private int fooCounter = 0;
    [BindableData(1)]
    public int FooCounter
    {
      get => fooCounter;
      set { if (fooCounter != value) { fooCounter = value; OnPropertyChanged(); CheckFooBar(); } }
    }

    private int barCounter = 0;
    [BindableData(2)]
    public int BarCounter
    {
      get => barCounter;
      set { if (barCounter != value) { barCounter = value; OnPropertyChanged(); CheckFooBar(); } }
    }

    [BindableAction(100)]
    private void IncrementFoo()
    {
        FooCounter++;
    }

    [BindableAction(101)]
    private void IncrementBar()
    {
        BarCounter++;
    }

    private void CheckFooBar()
    {
        IsFooBar = (FooCounter + BarCounter) % 5 == 0;
    }
}
```

Voila! We have now written the code that will publish updates at runtime and notify any subscribers that they should update. 

Now you may be thinking to yourself "but why did we include those numbers inside the `BindableData` and `BindableAction` attributes?". The answer comes back to that idea of a shared understanding between publishers and consumers of data. The answer is that we need an identifier that is unique to that piece of data within the source (aka View Model), so we assign an arbitrary integer to be the ID. We could use the name of the data item (and that is where the framework started way back when), but then we would break the UI any time we wanted to rename a variable, and that's no good!

So when you're creating data items, simply choose a value and trust that the compiler will yell at you if you have a conflict (which it will).

Let's move onto our last steps for the publishing side of the framework. The next step happens purely in the Inspector.

2. Click the button in the Inspector that says "Update Data Record" (seen below)
3. We can now consume the data!

#### Data Records

Data Records are a Scriptable Object that lives on disk and **is** the shared understanding between publishers and consumers. It contains a list of information about all bindable data items within a source, including their string names, integer IDs, as well as any other info passed along in the bindable attribute like a comment. They must be generated after making updates to the data contained within a source. Custom Inspector logic is already provided for all classes inheriting from the base View Models.


### Consuming Data

The workflow for consuming data happens entirely in the Inspector using a component called `DataBinder`. The `DataBinder` component is a wrapper around a list of Binders, which can be thought of as bite-sized pieces of code that know how to connect a specific type of data to a specific type of UI element. For example, let's look at `SpriteBinder`, which comes built-in to the framework. It's purpose is to take a data value of type `Sprite` and set it on an `Image` component.

```
using UnityEngine;
using UnityEngine.UI;
using MVVMDatabinding;

namespace MVVMDatabinding
{
    // Binds a Sprite property in the ViewModel to the sprite of a UnityEngine.UI.Image
    public class SpriteBinder : BaseBinder<Sprite>
    {
        [SerializeField]
        private Image targetImage = null;

        [SerializeField]
        private bool disableIfNull = true;

        protected override void OnDataUpdated(Sprite value)
        {
            if (targetImage != null)
            {
                targetImage.sprite = value;

                if (value == null && disableIfNull)
                {
                    targetImage.enabled = false;
                }
                else if (value != null && !targetImage.enabled)
                {
                    targetImage.enabled = true;
                }
            }
        }
    }
}
```

`SpriteBinder` inherits from `BaseBinder` and declares it is binding to type `Sprite`. This means most of the logic is already handled by the underlying base class, so all that's needed is to set `targetImage.sprite` to the incoming `value`. 

Brief Tangent about Debugging - 
Many Unity UI implementations end up repeating this type of code, which increases the likelihood of bugs popping up needing debugged. In this framework, it is very unlikely that a single instance of using `SpriteBinder` will break while other instances work, meaning you can quickly focus your debugging onto 1) checking that it's bound to the correct piece of data or 2) ensuring the correct data is populated by the source.

#### Custom Binders

Binders give you an incredible amount of power and flexibility to interpret data for presentation on the UI. While you could add a separate boolean to track whether a value is 0 or not, you could also write a special binder that sets Game Object visibility based on a comparison against a data value. You could also write a custom binder that interprets a data type that is specific to your project, like an enum or class, and then your own custom logic for how to interpret that type and make the UI change in accordance with it. 

Custom binders should inherit from `BaseBinder<T>` in most cases, though rarely they will also inherit from `BaseBinder` in order to interpret multiple data types like `FormattedStringBinder` does.

⚠️⚠️**WARNING**⚠️⚠️

Once you have created a custom binder, changing its name OR the name of the namespace and assembly it's contained within will break any existing UI bindings you have set up. Binders are stored as [SerializeReferences](https://docs.unity3d.com/6000.0/Documentation/ScriptReference/SerializeReference.html) inside the Game Object's YAML and SerializeReferences are sensitive to class, namespace, and assembly name changes.

If you absolutely must update one of those, there are high-risk ways to update the YAML, namely by doing a Find & Replace on the entire project looking for the name and changing it. This could break your UI and/or project, so do not go about it lightly.
